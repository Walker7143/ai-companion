# 记忆整理能力产品化 PRD

> 状态：建议立项
> 日期：2026-05-22
> 适用范围：`ai-companion` 主聊天、CLI/Gateway 命令面、管理后台记忆页

## 1. 背景

近期参考文章《你的OpenClaw开启“梦境”了吗？》把长期记忆整理包装为一个用户可理解的功能：

- 空闲时后台整理近期交互
- 分阶段筛选和提炼记忆
- 将高价值内容提升到长期记忆
- 产出人类可读的整理报告
- 提供显式开关、状态查看和手动触发命令

这篇文章的启发不在于底层技术突破，而在于它把“记忆系统”从一个内部机制，包装成了一个用户可感知、可理解、可控制、可纠错的产品能力。

对本项目而言，底层记忆能力已经明显强于文章中呈现的版本：

- 已有 `Working / Daily / Episodic / Semantic / Relationship / UserUnderstanding` 分层
- 已有 `MemoryPromptBuilder` 做按意图投影和预算控制
- 已有 `user_understanding.json` 作为可编辑的长期理解投影
- 已有主动连续性、self memory、记忆诊断、向量索引重建等能力

因此，本项目的核心问题不是“要不要做记忆系统”，而是：

**是否要把现有记忆系统进一步产品化，让用户真正知道它在记什么、为什么记、何时记、如何纠正。**

## 2. 产品经理结论

### 2.1 是否有必要沉淀

有必要。

但应沉淀为：

**“现有记忆系统的产品化包装与可观测化方案”**

而不是：

**“照抄 OpenClaw 梦境机制，重新发明一套记忆内核”**

### 2.2 为什么值得做

这件事有四个明确价值：

1. 提升用户理解成本

当前项目的记忆系统很强，但概念较重，用户很难直接理解 `daily memory`、`episodic recall`、`user_understanding projection` 这些内部术语。一个统一的产品概念能降低理解门槛。

2. 提升用户信任

用户更容易相信“我看到了你记住了什么，也能删掉不该记的东西”，而不是相信“系统会自己处理好记忆”。

3. 提升长期使用感知

多数用户对“这个 Bot 越聊越懂我”的感知，不来自底层 recall 指标，而来自几个非常直观的体验：

- 它能自然接住前面聊过的事
- 它不会机械重复旧档案
- 它能解释为什么会记住某些内容
- 它不会把低价值内容乱记一通

4. 提升运营与排障效率

当用户反馈“它怎么老记错”“它怎么突然不记得了”“它是不是又把我当资料库在读”，统一的记忆整理入口和报告机制会让诊断更直接。

### 2.3 为什么不能照搬原文方案

不能照搬，原因有三点：

1. 我们已经有更复杂的真实存储结构

文章里的 `MEMORY.md` / `DREAMS.md` 更像单一文件视角。本项目的长期真相源并不是一份 Markdown，而是多层结构化存储加投影文件。

2. 我们的重点是“像真人地使用记忆”，不是“展示记忆文件”

本项目已经明确区分“完整存储的记忆”和“本轮进入意识工作区的记忆”。产品层也应该延续这个原则，而不是让用户误以为所有记忆都应该直接进入对话。

3. 我们需要兼顾关系型陪伴体验

本项目不是通用 Agent 笔记工具，而是带人格、边界、关系态、主动连续性的 AI 伴侣。记忆整理能力必须服务于“更自然的相处感”，而不是把用户感受变成运维视角。

## 3. 产品定位

### 3.1 定位定义

“记忆整理”是 AI Companion 面向用户开放的一项后台能力。它会在不打扰聊天的前提下，对近期互动进行筛选、归纳和沉淀，把少量值得长期保留的内容纳入长期理解，并生成可查看、可纠正的整理结果。

### 3.2 推荐命名

内部方案名：

- 记忆整理
- 长期记忆整理

用户态包装名候选：

- 梦境
- 夜间整理
- 回忆整理

建议：

**对外可用“梦境”作为轻产品名，对内和文档统一用“记忆整理”作为正式能力名。**

理由：

- “梦境”有记忆整理的想象空间，传播友好
- “记忆整理”更准确，便于和现有系统架构对齐
- 未来即使用户态改名，技术和文档层仍可稳定复用“记忆整理”概念

## 4. 产品目标与非目标

### 4.1 目标

V1 目标：

1. 让用户知道该能力存在，并理解其作用
2. 让用户能开启、关闭、查看状态、手动触发
3. 让用户能看到“本次整理了什么、为什么整理、哪些进入长期层”
4. 让用户能删除或修正自动沉淀的记忆结果
5. 不重写底层记忆引擎，仅在现有能力上做产品层封装

### 4.2 非目标

V1 不做：

1. 不重构现有 Memory Engine 主体架构
2. 不把所有长期记忆改成 Markdown 文件
3. 不承诺“所有记忆都可自动完美整理”
4. 不在普通聊天中直接暴露底层记忆术语
5. 不把主动唤醒、提醒、梦境、人生事件混成一个大杂烩入口

## 5. 用户问题定义

当前用户在记忆相关体验上常见的真实问题是：

1. 我不知道它到底记住了什么
2. 我不知道它记错了能不能改
3. 我不知道为什么有时候它能接上，有时候又像忘了
4. 我不知道系统是不是把所有东西都塞进 prompt，才导致说话变怪
5. 我不知道记忆功能到底有没有在运行

这说明我们当前缺少的是：

- 统一产品概念
- 用户可见状态
- 可读结果输出
- 可纠错入口

## 6. 方案总览

### 6.1 核心产品思路

基于现有记忆架构，增加一层“用户可见的整理流程”：

1. 采集近期可整理的候选内容
2. 按规则筛除低价值、低置信、敏感不宜直提的内容
3. 输出本次整理摘要
4. 只把少量高价值结果提升到长期层
5. 将整理结果对用户可视化，并允许纠错

### 6.2 产品原则

1. 后台进行，不打扰聊天
2. 结果可看，但不强迫用户管理
3. 自动为主，手动纠错优先
4. 少量高价值沉淀，反对“全记住”
5. 以相处自然为目标，不以资料堆积为目标

## 7. 与现有能力的映射

本方案原则上复用现有模块，不重复造轮子。

| 产品概念 | 现有基础 |
| --- | --- |
| 近期可整理上下文 | `WorkingMemoryStore`、`DailyMemoryStore` |
| 共同经历沉淀 | `EpisodicStore` |
| 稳定事实与偏好沉淀 | `SemanticStore`、`user_understanding.json` |
| 关系变化承接 | `RelationshipStore`、relationship narrative |
| 主聊天使用控制 | `MemoryPromptBuilder`、Conscious Context 设计 |
| 状态诊断 | `/memory`、管理后台 diagnostics、记忆统计 |
| 用户纠错 | 删除自动事实、编辑 `user_understanding`、向量索引重建 |

这意味着 V1 可以不改记忆底座，只补产品入口、任务编排、报告展示和用户操作面。

## 8. 功能设计

## 8.1 功能一：能力开关

### 目标

让用户知道“记忆整理”是一个可控能力，而不是隐式黑箱。

### 设计

新增统一开关：

- 开启记忆整理
- 关闭记忆整理

支持入口：

- CLI / Gateway 命令
- 管理后台设置页

### 建议命令

用户友好别名：

- `/dream on`
- `/dream off`

正式别名可兼容：

- `/memory organize on`
- `/memory organize off`

说明：

不要求一定只保留一个命名。可以在用户态支持轻命名，在内部帮助文档与后台配置中使用正式能力名。

## 8.2 功能二：状态查看

### 目标

回答用户最常见的问题：

- 现在是不是开着
- 最近一次什么时候整理过
- 上次整理出了什么结果
- 当前有没有待整理内容

### 设计

新增状态视图，展示：

- 是否启用
- 最近一次整理时间
- 最近一次整理结果摘要
- 近期候选记忆数量
- 最近提升到长期层的条目数量
- 最近一次失败原因

### 建议命令

- `/dream status`
- `/memory organize status`

### 后台展示

在“记忆”页顶部新增“记忆整理状态卡片”。

## 8.3 功能三：手动触发

### 目标

让高级用户和调试场景可以主动触发一次整理，不必等待后台节奏。

### 设计

提供手动运行入口：

- `/dream run`
- 后台按钮“立即整理一次”

运行后给出简短结果：

- 本次扫描了多少候选
- 保留了多少
- 提升了多少
- 是否生成了整理报告

## 8.4 功能四：整理报告

### 目标

把“系统内部发生了什么”转成用户能理解的结果输出。

### 报告结构建议

每次整理后生成一份可读摘要，至少包括：

1. 本次记忆整理结论
2. 进入长期理解的内容
3. 仅保留为近期连续性的内容
4. 被忽略的内容类型说明
5. 用户可执行操作

### 示例结构

```text
本次记忆整理完成

我保留了这些长期有价值的信息：
- 你最近在忙 XX
- 你更喜欢我先共情再给建议

这些内容我暂时只当作近期上下文，不会长期记住：
- 今天临时提到的一次性事项

这些内容我没有长期保留：
- 低置信猜测
- 与关系无关的重复闲聊

如果有哪条不准确，你可以删除或手动修正。
```

### 设计原则

1. 不暴露底层技术术语
2. 不把用户当运维
3. 不说“我从数据库记住了”
4. 强调“我会自然使用，不会机械复述”

## 8.5 功能五：纠错与删除

### 目标

让用户对自动记忆有最终控制权。

### V1 范围

支持：

- 删除某条自动沉淀结果
- 清空本次整理新增的自动事实
- 跳转查看用户理解文件或后台对应卡片

### 暂不要求

- V1 不做复杂的逐条人工审核工作流
- V1 不做完整的“草稿箱审批”机制

## 8.6 功能六：诊断修复

### 目标

当用户觉得“好像没记住”“记忆乱了”“最近说话怪”，能快速给出自检入口。

### 设计

提供诊断命令：

- `/dream doctor`

输出内容包括：

- 记忆引擎是否启用
- 索引是否可用
- 用户理解文件是否存在
- 最近一次整理是否成功
- 是否建议重建索引或重启运行时

这里重点不是做复杂修复，而是把现有诊断能力统一收口。

## 9. 交互设计

## 9.1 命令面

建议首批命令：

- `/dream on`
- `/dream off`
- `/dream status`
- `/dream run`
- `/dream doctor`
- `/dream report`

其中：

- `report` 用于查看最近一次整理结果
- `status` 偏运行状态

## 9.2 管理后台

建议在现有“记忆”页增加一个“记忆整理”分区：

1. 顶部状态卡片
2. 最近一次整理摘要
3. 立即整理按钮
4. 最近整理历史列表
5. 本次新增长期记忆条目列表
6. 删除/修正入口

## 9.3 用户感知文案

需要统一口径：

- “我会在不打扰你的前提下，整理近期值得长期记住的内容。”
- “不是每句话都会长期记住。”
- “如果有记错，你可以删掉或修正。”
- “我会把记忆当作相处背景自然使用，而不是反复复述。”

## 10. 关键产品取舍

## 10.1 做“自动整理结果”，不做“全量记忆浏览器”

原因：

- 用户真正需要的是“我最近记住了什么”
- 不是所有人都愿意管理底层记忆资产
- 全量浏览器更适合后台和高级调试

## 10.2 做“少量高价值沉淀”，不做“所有内容都记住”

原因：

- 长期陪伴的关键是判断力，不是囤积量
- 低价值记忆进入长期层会损害自然度
- 过量沉淀会反过来推高 prompt 噪音

## 10.3 做“可解释与可纠错”，不做“强人工审批流”

原因：

- 人工审批流太重，会破坏陪伴产品的自然使用感
- 本项目更适合“自动为主，纠错兜底”

## 11. 成功指标

V1 建议关注：

1. 开启率

- 有多少用户在看到能力介绍后主动开启

2. 使用感知

- 用户是否更容易理解“为什么它会记得我之前说过的话”

3. 纠错效率

- 用户是否能快速删除错误自动记忆

4. 体验稳定性

- 因记忆而产生的“资料感”“机械复述感”反馈是否下降

5. 支持诊断效率

- 与“记忆失效/记错”相关的问题排查成本是否降低

## 12. 风险与约束

## 12.1 最大风险

如果包装做得太重，而底层实际行为仍然不稳定，会放大用户预期落差。

因此 V1 必须坚持：

- 少承诺
- 强状态
- 强纠错
- 强诊断

## 12.2 运行时边界

本项目很多真实数据源在 `~/.ai-companion/` 运行时目录，而不在仓库 `data/` 种子目录。产品文档、帮助文案和后台行为必须以运行时真相为准。

## 12.3 关系型产品边界

敏感记忆、关系变化、情绪经历不能简单做成“事实列表”。整理报告里必须避免过度直白地点名敏感内容，优先输出“记忆使用边界”和“关系理解变化”。

## 13. 分阶段落地建议

## Phase 1：产品入口收口

目标：

- 有统一命名
- 有开关
- 有状态
- 有手动触发

范围：

- CLI / Gateway 命令
- 后台状态卡片
- 基础 diagnostics 汇总

## Phase 2：整理报告

目标：

- 用户能看懂最近一次整理发生了什么

范围：

- 最近一次整理摘要
- 本次新增长期记忆列表
- 删除入口

## Phase 3：更强的解释与纠错

目标：

- 用户能理解为什么某些内容被提升、某些没有

范围：

- 理由标签
- 置信度/重要性可视化
- 更多纠错入口

## Phase 4：与主动连续性和关系叙事联动

目标：

- 让“记忆整理”不仅服务于存储，还服务于更自然的主动关心和关系承接

范围：

- 与 `daily memory`
- 与 `conversation continuity`
- 与 `relationship narrative`
- 与 `self_memory`

## 14. 最终结论

这篇文章对本项目最有价值的参考，不在于它的实现细节，而在于它做对了两件事：

1. 给复杂记忆系统一个用户能理解的产品名字
2. 给黑箱式长期记忆一个可见、可控、可纠错的交互闭环

因此，本项目建议沉淀该能力。

但沉淀方式应是：

**在现有记忆架构之上，做“记忆整理/梦境”的产品化包装、状态可视化、报告输出与纠错入口。**

而不是：

**重做一套以 Markdown 文件为中心的新记忆系统。**

从投入产出比看，这是一个值得做、且适合分阶段快速落地的产品项。
