# 记忆系统产品收敛方案 PRD

> 读者：架构师 / 产品经理 / 核心研发
> 日期：2026-05-22
> 文档目标：基于当前系统现状，重新定义记忆系统的产品边界、概念层次和后续优化方向，解决“能力越来越多但认知越来越乱”的问题

---

## 1. 文档结论

当前 AI Companion 的记忆系统**底层能力并不乱**，真正的问题在于：

1. 存储层、派生层、整理层、解释层都被平铺成了“记忆功能”
2. 多个能力都在回答“记住了什么”，但其实语义不同
3. 对用户、产品、研发来说，缺少一张统一的“总分层地图”

因此，本次 PRD 的目标不是新增一套记忆能力，而是：

**对现有记忆系统进行产品收敛，让系统更好讲清、更好管理、更好扩展。**

---

## 2. 背景与问题

### 2.1 当前系统现状

项目已经具备完整的多层记忆体系：

- `Working`
- `Daily`
- `Semantic`
- `Relationship`
- `Episodic`
- `UserUnderstanding`
- `Vector Index`
- `Rollup`
- `ConsciousContext`
- `Dreaming`
- `Memory Trust View`
- `Doctor`

从能力角度看，这是优势。

但从认知角度看，这已经带来一个明显问题：

**用户和团队成员会逐渐说不清每一层到底是什么、彼此是什么关系、哪些是真记忆、哪些只是辅助结构。**

### 2.2 当前最典型的困惑

1. `语义记忆` 和 `用户理解` 到底有什么区别？
2. `向量记忆` 是不是又存了一份？
3. `梦境整理` 是不是在重做记忆系统？
4. `记忆可信视图` 和 `梦境报告` 为什么看起来都像在解释“记住了什么”？
5. `Maintenance`、`Governor`、`Dreaming` 后续会不会变成三套整理逻辑？

这些不是使用者的问题，而是产品表达层缺少统一设计的问题。

---

## 3. 产品判断

### 3.1 这是不是一个“功能过多”的问题？

不是。

它本质上是一个：

**信息架构失衡问题**

即：

- 底层能力增长速度快
- 上层概念整理速度慢

### 3.2 这是不是一个“架构错误”的问题？

不是。

当前底层架构总体方向是对的：

- 有结构化真相源
- 有召回层
- 有意识工作区
- 有 prompt 预算控制
- 有用户纠错入口

问题不在于技术方向错了，而在于：

**产品层没有把这些能力重新组织成一套容易理解的分层。**

### 3.3 这次是否需要重构底层？

不需要。

本 PRD 明确主张：

**优先做产品收敛，不做底层重写。**

---

## 4. 本次产品目标

本次产品收敛要实现的不是“增加更多功能”，而是把现有功能重新讲清楚。

### 4.1 核心目标

1. 给记忆系统建立一套统一的产品分层
2. 让用户和团队成员都能快速理解每类能力的角色
3. 降低“为什么这么多记忆相关概念”的认知负担
4. 为后续自动整理、梦境调度、后台重构提供稳定的信息架构基础

### 4.2 非目标

1. 不重写 `MemoryEngine`
2. 不替换现有 `Semantic / Relationship / Episodic / Daily` 真相源
3. 不在本阶段新增复杂的新存储层
4. 不把所有问题都归结为“换个名字就行”

---

## 5. 产品原则

### 原则 1：用户不该看到“所有内部层次的并列名词”

内部为了工程清晰可以有多层，但对外不应该把每层都当作平级产品概念展示。

### 原则 2：真相源和派生层必须明确区分

哪些地方保存“真正的事实”，哪些地方只是为了召回、解释、压缩、展示而存在，必须清晰。

### 原则 3：状态视图和事件报告要分开

“系统现在稳定记得什么”和“本次整理刚刚做了什么”是两件事，不能混成一个页面逻辑。

### 原则 4：整理能力必须是产品层编排，而不是又一套底层记忆体系

`Dreaming` 应该是对现有记忆系统的产品化整理能力，而不是新的长期记忆真相源。

### 原则 5：后续扩展必须遵守总分层

以后再增加新能力时，必须先判断它属于哪一层，再决定放到哪里。

---

## 6. 新的产品分层模型

为了收敛现有概念，本 PRD 建议将整个记忆系统统一拆成四层。

---

### 6.1 第一层：权威记忆层

这是“真正保存内容”的层。

包含：

- `Working`
- `Daily`
- `Semantic`
- `Relationship`
- `Episodic`

#### 产品定义

这层是系统的底账，是“Bot 真正持有什么信息”的基础。

#### 设计要求

- 所有长期能力必须最终回落到这层或明确依附于这层
- 不允许新增平行的“第二套长期真相源”

---

### 6.2 第二层：记忆投影与召回层

这是“为了更好读取、更好召回、更好控制 prompt”而存在的派生结构。

包含：

- `UserUnderstanding`
- `Vector Index`
- `Rollup`
- `ConsciousContext`

#### 产品定义

这层不拥有最终真相，只负责把真相源转成更适合使用、展示或推理的形式。

#### 设计要求

- 明确标注为“投影层 / 派生层 / 读模型”
- 后台展示时避免让用户误以为是第二份记忆库

---

### 6.3 第三层：整理与运维层

这是“围绕记忆做动作”的层。

包含：

- `Dreaming`
- `Maintenance`
- `Rebuild Vector`
- `Doctor`

#### 产品定义

这层负责整理、维护、恢复、排障，但不负责定义长期真相本身。

#### 设计要求

- `Dreaming`：面向用户的整理任务
- `Maintenance`：底层 housekeeping
- `Doctor`：诊断入口
- `Rebuild Vector`：索引级运维动作

必须避免它们后续互相长成平行体系。

---

### 6.4 第四层：解释与展示层

这是“给人看懂系统行为”的层。

包含：

- `Memory Trust View`
- `Dreaming Report`
- `memory_prompt_diagnostics`

#### 产品定义

这一层不新增记忆，只负责解释：

- 系统现在持有什么
- 本次整理做了什么
- 本轮 prompt 为什么会带入这些记忆

#### 设计要求

- `Trust View` = 当前状态解释
- `Dreaming Report` = 单次事件解释
- `Diagnostics` = 研发/调试解释

---

## 7. 概念重定义

### 7.1 `UserUnderstanding`

现状问题：

它很容易被理解成“另一份长期记忆库”。

新定义：

**用户长期理解投影**

解释：

- 它不是全部记忆真相
- 它是用户可编辑、系统可投影、主聊天可按意图引用的长期理解层

产品层建议：

- 后台和文档中减少“用户理解文件”这种中性叫法
- 增加“长期理解投影”的解释

---

### 7.2 `Vector Memory`

现状问题：

“向量记忆”容易让人误以为系统又存了一份完整记忆。

新定义：

**统一语义召回索引**

解释：

- 它不是记忆真相源
- 它是为了语义召回而建立的索引层
- 可以重建
- 丢了也不等于记忆真相丢了

产品层建议：

- 后台里尽量用“向量索引”而不是“向量记忆”

---

### 7.3 `Dreaming`

现状问题：

如果不控制边界，后续很容易被做成第二套记忆巩固系统。

新定义：

**面向用户的记忆整理任务**

解释：

- 它是产品层编排能力
- 它从现有记忆层收集候选
- 它生成用户可读报告
- 它不拥有长期真相源

---

### 7.4 `Trust View`

现状问题：

和 Dreaming Report 都在讲“系统记住了什么”，容易混。

新定义：

**当前记忆状态可信视图**

解释：

- 它不是历史整理报告
- 它是“现在系统稳定持有什么”的横截面

---

## 8. 产品信息架构改造

### 8.1 后台信息架构

当前 Memory 页面建议调整为四组：

1. 当前与短期
   - Working
   - Daily

2. 长期与关系
   - Semantic
   - Episodic
   - Relationship
   - User Understanding

3. 索引与投影
   - Vector Index
   - Rollup
   - Prompt Diagnostics

4. 整理与解释
   - Dreaming
   - Trust View
   - Doctor

### 8.2 页面目标变化

Memory 页面不再只是“记忆列表”，而是：

**记忆系统操作与解释中心**

---

## 9. 命令体系收敛

### 9.1 `/memory`

负责：

- 查看系统当前记忆状态
- 理解存量记忆和运行状态

不负责：

- 表达单次整理事件

### 9.2 `/dream`

负责：

- 整理任务开关
- 手动触发
- 查看最近整理结果
- 删除最近整理新增项
- 整理相关诊断

### 9.3 `doctor`

负责：

- 统一诊断和修复建议

原则：

不要继续在命令层新增更多平级新概念。

---

## 10. 产品边界要求

### 10.1 Dreaming 与 Maintenance 的边界

建议明确规定：

#### `Maintenance`

负责：

- 底层轻量维护
- 归档
- 衰减
- rollup
- 索引刷新

不负责：

- 用户态整理报告
- 用户态整理入口

#### `Dreaming`

负责：

- 用户态整理任务
- 整理报告
- 整理状态
- 最近整理结果删除
- 用户理解层面的解释

不负责：

- 底层长期真相定义

---

### 10.2 Trust View 与 Dreaming Report 的边界

#### Trust View

回答：

- 系统现在稳定持有什么记忆
- 哪些高置信
- 哪些待确认
- 哪些已纠正或归档

#### Dreaming Report

回答：

- 这次整理刚刚做了什么
- 这次新增沉淀了什么
- 这次没沉淀什么

原则：

一个是“状态”，一个是“事件”，必须分开。

---

## 11. 分阶段落地

### Phase 1：概念收敛

目标：

- 文档统一
- 页面分层明确
- 术语统一

交付：

- 新版产品文档
- 总架构图
- Memory 页面信息架构重组设计

### Phase 2：产品入口收敛

目标：

- `/memory`、`/dream`、`doctor` 语义稳定
- 后台入口层次稳定

交付：

- 命令说明整理
- 后台入口归位

### Phase 3：自动执行前的边界固化

目标：

- 在接 Dreaming 自动执行前，先明确它和 Maintenance 的分工

交付：

- 自动整理触发策略文档
- 运行时边界说明

### Phase 4：再接自动执行

目标：

- 在不增加系统认知混乱的前提下，增加 Dreaming 自动运行

---

## 12. 最终产品结论

当前记忆系统的问题不是“功能做太多了”，而是：

**同一套系统里的不同层次，没有被重新组织成一个清晰的产品模型。**

因此本次优化版产品文档的核心主张是：

1. 保留现有底层能力
2. 对现有概念重新分层
3. 把真相源、派生层、整理层、解释层彻底拆开讲
4. 用统一的信息架构解决“为什么越来越多、越来越难懂”的问题

一句话总结：

**现在该做的不是再造一套记忆系统，而是把现有记忆系统重新讲明白、重新排清楚。**
